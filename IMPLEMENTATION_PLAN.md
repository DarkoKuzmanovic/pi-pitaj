# pi-pitaj expansion ideas

Source: 2026-05-31 discussion comparing `@juicesharp/rpiv-advisor` with local `pi-pitaj`.

## Goal

Keep `pitaj` as the primary controlled sidecar-consultation tool, but borrow the best ideas from `rpiv-advisor` without dumping the full noisy conversation branch by default.

## Ideas

### 1. Snapshot consult mode

Add a mode that automatically builds a compact curated session snapshot and sends it to a selected model.

Possible shape:

```ts
pitaj({
  model: "opus",
  mode: "risk-check",
  question: "Review the current decision point from the provided session snapshot.",
  context: "<auto-generated compact session snapshot>"
})
```

Requirements:

- Prefer a compact snapshot over full branch serialization.
- Include current decision point, recent user request, relevant tool results, open risks, and active plan.
- Exclude unrelated earlier discussion and raw noisy tool output.
- Keep existing `maxContextChars` safeguards.

### 2. `/pitaj config` UI

Add an interactive model/settings picker similar to `rpiv-advisor`'s `/advisor` selector.

Should configure:

- default model
- low-risk auto route model
- high-risk auto route model
- default mode
- default brevity/output length
- optional reasoning effort where supported

Persist settings safely in the extension settings file or a dedicated config file.

### 3. Zero-parameter advisor-style shortcut

Consider a convenience command/tool for escalation when the agent knows it needs a stronger opinion but does not want to hand-write a prompt.

Possible command names:

```text
/pitaj advise
/pitaj snapshot
/pitaj escalate
```

Possible tool shape:

```ts
pitaj_advisor()
```

Design constraint:

- Do **not** blindly serialize and send the whole session by default.
- Use the curated snapshot builder from idea #1.
- Make the output clearly advisory, not authoritative.

### 4. Optional full-branch mode, behind explicit opt-in

If we ever want parity with `rpiv-advisor`, add an explicit full-branch mode.

Possible command:

```text
/pitaj advise --full-branch
```

Guardrails:

- Warn about token cost and noisy/sensitive context.
- Require explicit user/action confirmation or config flag.
- Keep disabled by default.

## Why not just install `rpiv-advisor`?

`rpiv-advisor` is convenient because `advisor()` has zero parameters and forwards the whole current conversation branch to a configured reviewer model. That is useful for "review everything so far" escalation.

For this setup, `pitaj` is better as the default primitive because it is:

- explicit about the question
- bounded in context size
- cheaper and more predictable
- better aligned with PMTI artifacts and focused decision reviews
- locally owned and easier to adapt

## Initial implementation order

1. Add snapshot-builder helper and unit tests.
2. Add `/pitaj snapshot` or `/pitaj advise` command using existing consult path.
3. Add `/pitaj config` UI after the snapshot path works.
4. Only then consider optional full-branch mode.
