## Goal

Make the easy path easier without changing the consult or context-capture boundary. M4 adds:

1. A zero/low-parameter **advisory shortcut** that wraps the existing M1 curated snapshot builder and clearly labels its output as advisory.
2. A **`/pitaj auto`** slash command giving risk-hinted auto-routing parity on the command surface (auto-routing currently exists only on the tool API).
3. A **test split** of the monolithic `helpers.test.ts` into focused suites for faster, clearer runs.

Every change wraps an existing path. M4 adds no new context-capture source, no persistence, no token/cost accounting, and no sidecar tools.

## User-visible / operational value

After M4, a worker (especially MiniMax) can run a single advisory command to get a second opinion using the already-bounded curated snapshot, without remembering the snapshot syntax. Interactive users gain `/pitaj auto --risk high|low` parity with the tool-API auto router. Contributors get targeted test files instead of one ~1000-line suite.

### Q2 — `/pitaj advise` affordance shape

**Finding:** ⚠️ **WARNING — resolved.** Orchestrator chose **Option 1: zero-flag only.** `/pitaj advise [question]` accepts only a bare question. `--mode`, `--brevity`, `-c`, and model-as-first-arg are all rejected with a clear error. This creates a genuine ergonomic distinction from `snapshot` and prevents scope creep.

### Q3 — Snapshot budget threshold

**Finding:** ✅ **NOTE — sharing the M3 snapshot threshold is correct.** M4-T2 packet must set `hasSnapshot: true` in usage recording for advisory consults so they count toward the existing 5-consult snapshot warning.

### Q4 — Oracle-lane T2 bundled with Review/Fast T1/T3

**Finding:** ✅ Bundling is acceptable with Batch A/B structure. ⚠️ **WARNING — M4-T2 packet must include explicit instruction:** "Re-read `index.ts` and `helpers.ts` in full before making any edit. Do not assume line positions from the M4-T1 diff — T1 will have shifted them. Treat the post-T1 state as the baseline for all T2 edits."

### Q5a — `classifySpecialCommand` prefix gap (BLOCKER — fixed in-plan)

**Finding:** 🚨 The current `classifySpecialCommand` uses a `switch` that only matches exact normalized strings. `/pitaj auto --risk high is this safe?` normalizes to `"auto --risk high is this safe?"` — the switch returns `"none"`, the input falls through to `parseCommandArgs`, and `"auto --risk high is this safe?"` becomes the question text with no risk parsing. **Silent failure.**

**Fix applied to M4-T1 acceptance criteria:** Add `if (normalized === "auto" || normalized.startsWith("auto ")) return "auto";` BEFORE the switch, following the existing `config` pattern. Do NOT use a bare `case "auto":` in the switch. Same guard applies to `advise` in M4-T2.

### Q5b — Alias collision

**Finding:** ⚠️ **WARNING — `auto` and `advise` are now reserved subcommand names.** Any user with `"auto": "some/model"` or `"advise": "some/model"` in `settings.aliases` loses that alias silently because `classifySpecialCommand` intercepts before alias resolution. README and `usageText()` must list these as reserved names.

### Q5c — Snapshot dispatch pattern

**Finding:** 📝 **NOTE — `"snapshot"` in `SpecialCommand` type is currently dead code.** The dispatch never checks `specialCommand === "snapshot"`; the snapshot branch is driven by `parseSnapshotCommandArgs()`. When adding `auto` and `advise`, follow one pattern consistently. Recommendation: use `classifySpecialCommand` check in the dispatch chain for all subcommands (not dedicated parsers), keeping `parseSnapshotCommandArgs` only for the snapshot context building. Not a blocker.

### Risks remaining after fixes

- **Implementation scope creep on `advise`:** If the executor decides `advise` should support flags "for completeness," it recreates `snapshot` with a different name. The task packet must enumerate exactly which inputs are rejected.
- **T1 and T2 touching the same files:** If T2 starts before T1's reviewer signs off, line anchor corruption is likely. The T2 packet must require reading the committed diff from T1 before starting.
- **`usageText()` help string:** Both T1 and T2 must update it; they'll conflict if worked on simultaneously. Assign `usageText()` updates to whichever batch lands last (T2, since T4 is the doc task).
