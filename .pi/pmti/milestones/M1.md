**Oracle review:** required before task packet preparation because this milestone introduces automatic session/context capture.

## Goal

Add a safe snapshot-based consultation path that can automatically build a compact, curated session snapshot and send it through the existing `pitaj` consult flow without dumping the full conversation branch by default.

## User-visible / operational value

After M1, an agent or human can ask `pitaj` for a second opinion on the current decision point with relevant bounded context included automatically. The sidecar answer remains advisory, tool-less, and explicit about what context was provided.

## Scope

### In

- Define a snapshot contract for exactly which context categories may be included.
- Implement a deterministic snapshot builder helper with unit tests.
- Include, where safely available: current decision point/question, recent user request, active plan/tasks, relevant recent tool-result summaries, and open risks/limitations.
- Exclude unrelated older discussion, raw noisy tool output, secrets, and full-branch serialization.
- Enforce existing `maxContextChars` safeguards and add category/provenance markers before truncation.
- Wire snapshot context into the existing `consultModel()` path without changing normal explicit `context` behavior.
- Add one command surface: `/pitaj snapshot` as the M1 command name. `/pitaj advise` and `/pitaj escalate` remain later aliases/shortcuts unless explicitly approved in a later milestone.
- Update `/pitaj help`, README, and tests for the snapshot path.

### Out

- Full-branch capture or `--full-branch`; this belongs to M5.
- `/pitaj config` UI; this belongs to M2.
- Budget/usage tracking; this belongs to M3.
- Zero-parameter advisor tool registration such as `pitaj_advisor()`; this belongs to M4 if M1 proves the snapshot builder.
- Giving the sidecar model tools.
- Broad persistent consultation history.
- Adding `autoContext` or equivalent snapshot opt-in parameter to the `pitaj` tool schema (`PitajParams`); this belongs to M4 after the snapshot builder is proven.

## Acceptance criteria

- Existing explicit `pitaj` tool calls and `/pitaj <alias> ...` command behavior remain backward compatible.
- A pure snapshot builder exists with deterministic tests for included categories, missing categories, provenance labels, and truncation.
- Snapshot output clearly states that the sidecar has no tools and only sees the snapshot.
- Snapshot context is bounded by configured/requested `maxContextChars` and marks truncation or omission.
- `/pitaj snapshot <question>` calls the existing consult path with generated context.
- `/pitaj snapshot` rejects or prompts for an empty question consistently with existing `/pitaj` behavior.
- Result details or displayed metadata indicate snapshot usage and included/omitted categories without bloating the answer.
- README and `/pitaj help` document the snapshot command and warn that it is not full-branch review.
- `npm test` passes from `/home/quzma/.pi/agent/extensions/pi-pitaj`.
- Pre-code oracle review findings are recorded in this milestone before `pmti-task` packet preparation.

## Phase-boundary health

M1 leaves the project in a coherent intermediate state: `pitaj` still works as a focused sidecar consult, while snapshot-based convenience is available through one bounded command. Later milestones can add config UI, budgets, advisor shortcuts, or full-branch opt-in without replacing the snapshot builder or changing the core consult contract.

## Constraints and invariants

- Sidecar models remain tool-less and must not claim they inspected files unless the snapshot or caller-provided context contains the excerpt.
- Snapshot capture must be reactive and bounded, not predictive or broad vault/session scraping.
- Do not send full raw transcript or raw large tool output by default.
- Do not include secrets, API keys, auth headers, absolute local paths beyond already-visible project paths, or unbounded command output.
- Preserve TypeScript style: no `any`, no non-null assertions, type-only imports where applicable.
- Preserve current test command unless a task explicitly changes it: `npm test`.
- Use existing settings parsing and `maxContextChars` behavior where possible instead of creating a second config system.

### M1-T1 — Define snapshot contract and data model

**Implementation status:** completed 2026-06-01; see `.pi/pmti/project/session-log/2026-06-01-004.md`.
**Depends on:** none
**Blocks:** M1-T2, M1-T3, M1-T4, M1-T5
**Can run in parallel with:** none; this fixes the context boundary.

**Implementation context:** `helpers.ts`, `index.ts`, `README.md`, `.pi/pmti/project/decisions.md`, this milestone file.

**Acceptance:**

- Defines `SnapshotCategory`/metadata shape for categories such as `question`, `recent-user-request`, `active-plan`, `tool-results`, and `risks`.
- Defines category ordering and omission behavior when data is unavailable.
- Defines provenance labels shown in generated context.
- Records the concrete provenance label format as a string template, for example `[snapshot:tool-results — 3 results, 2048 chars]`, so M1-T2 tests can assert on the exact format.
- Classifies each snapshot category by its capture mechanism and omit-when-unavailable behavior:
  - `question`: caller-provided command parameter; required.
  - `recent-user-request`: `ctx.sessionManager.getEntries()` filtered to last N user `SessionMessageEntry` entries, bounded to `maxContextChars`; omit if none found.
  - `tool-results`: in-extension ring buffer populated by pre-registered `pi.on("tool_execution_end", ...)` hooks; omit if the ring buffer is empty.
  - `active-plan` and `risks`: caller-provided context or extension-appended `CustomEntry` only — not derived from session branch or general session message content; omit by default.
- Defines truncation policy: category-level trimming first where practical, then whole-snapshot `maxContextChars` enforcement with an explicit marker.
- Defines the M1 command name as `/pitaj snapshot` and leaves `/pitaj advise`/`escalate` out of scope.
- Extends the `SpecialCommand` type in `helpers.ts` to include `"snapshot"` and updates `classifySpecialCommand()` to return `"snapshot"` when the normalized input is `"snapshot"`, so M1-T4 command wiring cannot silently fall through to normal question parsing.
- Records any new durable decision in `.pi/pmti/project/decisions.md` only if implementation discovers a real tradeoff.
