**Status:** completed 2026-06-01
**Planned:** 2026-06-01

## Goal

Add a safe `/pitaj config` path that lets a human inspect and edit `pi-pitaj` settings without making `settings.json` less recoverable, less manually editable, or less compatible with existing `pitaj` and `/pitaj snapshot` behavior.

## User-visible / operational value

After M2, the operator can run `/pitaj config` to see the effective pitaj configuration and, in interactive UI mode, update common routing/output defaults through guided prompts instead of hand-editing JSON. Agents still use the existing `pitaj` tool contract; M2 improves operator configuration, not sidecar capability.

## Scope

### In

- Define the durable settings contract for M2, including backward-compatible fields for:
  - `defaultModel`
  - low-risk auto-route model alias/reference
  - high-risk auto-route model alias/reference
  - `defaultMode`
  - `defaultBrevity`
  - `maxContextChars`
  - `maxOutputChars`
  - aliases
- Resolve the known `maxOutputChars` schema/runtime ambiguity by making the loaded setting affect default output length in a documented, tested order.
- Add safe settings read/write helpers with deterministic tests.
- Add `/pitaj config` summary behavior that works in non-interactive contexts.
- Add an interactive UI flow for common settings when `ctx.hasUI` is true, using Pi dialog APIs or a compact custom UI only if warranted by implementation.
- Preserve malformed-file fallback behavior and avoid overwriting malformed `settings.json` without explicit confirmation or a recovery path.
- Keep `settings.json` manually editable, formatted, and inspectable.
- Update README and `/pitaj help` for `/pitaj config`.

### Out

- Budget/usage tracking; this belongs to M3.
- Advisor-style shortcuts; this belongs to M4.
- Full-branch mode; this belongs to M5.
- Sidecar tools or changes to the sidecar no-tools boundary.
- Replacing direct aliases, `model: "auto"`, or existing slash-command behavior.
- Persisting API keys, secrets, auth headers, or provider credentials.
- Editing Pi global provider/model configuration outside this extension's `settings.json`.
- A large bespoke TUI if built-in `ctx.ui.select`, `ctx.ui.input`, `ctx.ui.confirm`, or a small `SettingsList`-style component is enough.

## Acceptance criteria

- Existing explicit tool calls, `model: "auto"`, `/pitaj <alias> ...`, and `/pitaj snapshot ...` remain backward compatible.
- Settings contract is documented in code/tests and README.
- `maxOutputChars` has a single tested precedence order, recommended as: request override > settings value > brevity default.
- Auto-route low/high targets are configurable through settings while preserving current defaults (`gpt` for low/default and `opus` for high/risk-check unless changed).
- `/pitaj config` without UI returns a compact summary plus instructions, not a broken prompt.
- `/pitaj config` with UI can update default model, low/high auto-route model, default mode, default brevity, and max context/output lengths.
- Alias editing is either safely supported or explicitly deferred with a summary/instructions path; no partial alias editor is shipped ambiguously.
- Settings writes are safe: validate before writing, preserve aliases, write formatted JSON, avoid secrets, and do not silently overwrite malformed files.
- README and `/pitaj help` document `/pitaj config` and its limits.
- `npm test` passes from `/home/quzma/.pi/agent/extensions/pi-pitaj`.
- A fresh-context reviewer checks the concrete implementation before close-out.

## Phase-boundary health

M2 leaves the project in a coherent state: configuration becomes easier and safer without changing the core consultation model. Even if later M3/M4/M5 features are never implemented, the extension remains useful with explicit consults, auto-routing, snapshot consults, and operator-editable settings.

## Constraints and invariants

- Preserve TypeScript style: no `any`, no non-null assertions, type-only imports where applicable.
- Use `StringEnum`/TypeBox only for public tool schemas; do not add config-specific fields to `PitajParams` unless explicitly accepted by a later milestone.
- Keep settings defaults backward compatible with existing `settings.json` files.
- Do not write or store secrets.
- Do not mutate `settings.json` if parsing fails unless the user explicitly confirms replacing or recovering it.
- Prefer pure helper functions for settings normalization/serialization so tests do not need a live Pi UI.
- UI code must degrade gracefully when `ctx.hasUI` is false.
- Manual edits remain first-class: generated JSON should be readable and stable.

## Tasks

### M2-T1 — Define config contract and settings semantics

**Depends on:** M1 complete
**Blocks:** M2-T2, M2-T3, M2-T4, M2-T5
**Can run in parallel with:** none; this fixes the persisted settings contract.

**Implementation context:** `helpers.ts`, `index.ts`, `settings.json`, `README.md`, `.pi/pmti/project/watch.md`, this milestone file.

**Acceptance:**

- Defines exact M2 settings fields and defaults, including a backward-compatible auto-route settings shape.
- Decides and tests `maxOutputChars` precedence: request override > settings value > brevity default.
- Documents how invalid fields are ignored or repaired.
- Documents malformed-file behavior: fallback for read, no silent overwrite on write.
- Defines the three file states for write safety: file does not exist (offer to create), file exists and parsed cleanly (validate + confirm + overwrite), and file exists but malformed (refuse overwrite, show warning, and give manual recovery path unless an explicit confirmed recovery flow is implemented).
- Specifies the detection mechanism for write safety: either extend `LoadedSettings` with `fileState: "not-found" | "loaded" | "malformed"` or re-check `existsSync(SETTINGS_PATH)` and parse state at write time; record the chosen approach as a durable decision.
- Updates `AutoRouteResult.alias` from the hardcoded `"gpt" | "opus"` union to `string` so configurable auto-route targets do not require unsafe casts.
- Updates `resolveAutoRoute()` to read low/high target alias names from new settings fields while preserving defaults of `"gpt"` and `"opus"`.
- Decides whether absent numeric fields in `settingsFromUnknown()` remain fallback-populated or return `undefined` so `request > settings > brevity` precedence behaves intentionally.
- Records a durable decision in `.pi/pmti/project/decisions.md` if the field shape or write behavior involves a real tradeoff.

1. **Batch A — `M2-B1-config-core`:** covers M2-T1 through M2-T3 (settings contract, safe settings helpers/runtime semantics, and `/pitaj config` summary/non-UI fallback).

This replaces five per-task implementation/review loops with two bundled implementation/review loops while preserving the M2 dependency boundary before persistent interactive writes.
