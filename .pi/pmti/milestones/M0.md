# M0 — MiniMax-friendly auto-routing contract

**Status:** completed
**Planned:** 2026-05-30
**Source:** PMTI initialization after discussion of making `pitaj` easier for MiniMax to use proactively.

## Goal

Make the existing `pitaj` tool easy for MiniMax to call correctly by adding a backward-compatible auto-routing contract that lets the extension choose GPT versus Opus from explicit, compact request hints.

## Scope

### In

- Add a typed routing hint such as `topic` or `risk` to the tool request shape.
- Add `model: "auto"` support while preserving all existing alias and explicit `provider/model` calls.
- Add mode suggestion or mode-default guidance tied to the same topic/risk hints, without hiding the final mode sent to the sidecar.
- Implement deterministic routing rules for the first policy version:
  - GPT for concrete debug, API/tool uncertainty, localized critique, syntax/idiom checks, and fast implementation-plan sanity checks.
  - Opus for architecture/risk-check decisions, ambiguous intent, safety/policy/tone issues, high-cost-to-reverse changes, migrations, security/data-integrity concerns, and arbitration when MiniMax/GPT disagree.
- Make auto-routing rationale visible in result details without bloating the user-facing answer.
- Update `promptGuidelines` so MiniMax has a concise trigger rubric for `pitaj` and knows explicit `gpt`/`opus` overrides remain available.
- Update README/help text and tests for the new schema and routing helpers.

### Out

- Automatic context capture from session history, last tool output, diffs, or errors. That belongs to M1.
- Per-task budget tracking or warnings. That belongs to M2.
- Registering separate `pitaj_auto`, `pitaj_gpt`, or `pitaj_opus` tools. That belongs to M3 unless M0 proves the single-tool schema is too hard for MiniMax.
- Giving the sidecar model tools.
- Editing MiniMax model prompts outside this extension as part of M0.

## Acceptance criteria

- Existing tests pass with current alias/default behavior unchanged.
- New tests cover `model: "auto"`, topic/risk routing, manual override precedence, and unknown/invalid hint handling.
- New tests cover mode suggestion/default behavior where it is part of the M0 contract.
- The public tool schema describes auto-routing in terms MiniMax can choose reliably.
- Result details include the resolved model and, for auto-routing, the routing reason or selected route.
- README and `/pitaj help` document at least one direct alias call and one auto-routed call.
- `pitaj` promptGuidelines explicitly say when to use `pitaj`, when to prefer `pitaj` with GPT, when to prefer Opus, and when not to consult at all.

## Constraints and invariants

- Backward compatibility is mandatory for existing tool calls and `/pitaj` command syntax.
- Auto-routing must be deterministic and testable; no hidden model calls just to choose a model.
- Sidecar models remain tool-less and must not claim they inspected files unless context was provided.
- Context remains caller-supplied in M0.
- No hardcoded secrets, provider keys, or account assumptions.
- TypeScript style remains strict: no `any`, no non-null assertions, type-only imports where applicable.
- Validation command remains `npm test` from the extension root unless project scripts change intentionally.

## Oracle review

**Status:** recorded; approved; operator decisions accepted in `M0-b` and `M0-c`
**Reviewer:** `oracle-fresh` (`47092ea2-58d6-4489-8885-abda0a66f289`)
**Date:** 2026-05-30
**Artifact:** `context-build/oracle-m0-review.md`

### Findings and responses

**Blockers / required clarifications before task packet:**

1. `model: "auto"` must be intercepted before `resolveModelRef()`, because `resolveModelRef("auto", settings)` throws today. Oracle recommends a new pure helper `resolveAutoRoute()` in `helpers.ts`, called by `consultModel()` before `resolveModelRef()`, returning an alias plus routing metadata.
2. The public hint schema must be locked before implementation. Oracle recommends `risk?: "low" | "high"` rather than a larger `topic` enum, with mode-based fallback when risk is omitted.

**Non-blockers recorded for task packets / close-out:**

- `context-build/pitaj-product-context.md` uses milestone numbering that differs from PMTI; PMTI files under `.pi/pmti/` are authoritative for scope.
- `mesh-review/synthesis.md` contains stale findings for quoted `-c` context, `/pitaj check` API-key checking, and the `mimo` alias; current code already resolves those items.
- `PitajResultDetails` should explicitly name auto-route metadata fields such as `routingReason`, `autoRouted`, and `autoSuggestedMode`.
- `settings.maxOutputChars` is a pre-existing schema/runtime ambiguity and should be noted during M0-T5 or handled as follow-up.

**Verdict:** M0 is architecturally approved. `M0-T1` is ready for `pmti-task` preparation using the confirmed `risk?: "low" | "high"` schema and bundled `mm` alias decision.
