## Goal

Make `pitaj` consults understandable, auditable, and hard to overuse accidentally by improving the consultation result presentation, tracking lightweight in-session usage, surfacing budget warnings, and adding a usage summary command.

## User-visible / operational value

After M3, the `pitaj` tool response block should explain what happened without forcing the user to inspect hidden result details: which route/model answered, whether context was supplied, whether snapshot metadata applied, whether the sidecar had tools, and whether usage is approaching an advisory budget.

The result stays answer-first. Observability should clarify consultation behavior, not bury the answer in telemetry.

## Scope

### In

- Improve the displayed `pitaj` consultation result block.
- Add a pure/tested presentation helper that renders answer-first output plus compact metadata.
- Show route/model/mode metadata for auto-routed and explicit consults where available.
- Show snapshot/manual/no-context metadata where available.
- Clearly state that the sidecar has no tools/file access when context might otherwise be mistaken for real inspection.
- Add a lightweight in-memory usage event model for the current Pi session.
- Add advisory budget/warning semantics for low-risk/GPT-style, high-risk/Opus-style, and snapshot consults.
- Add `/pitaj usage` or equivalent summary command, with optional manual reset if implementation keeps semantics simple.
- Surface budget warnings in the result block after the presentation foundation exists.
- Update README, `/pitaj help`, and tests.

### Out

- Hard blocking of consults after budget thresholds.
- Persistent cross-session usage history.
- Pricing/cost estimation or token-accurate accounting.
- Automatic task detection beyond explicit session-scoped counters and optional manual reset.
- Full-branch capture or `--full-branch`; this remains M5.
- Advisor-style shortcuts such as `/pitaj advise`; this remains M4.
- Sidecar tools or any change to the sidecar no-tools boundary.
- Provider credential/API key persistence.
- Direct `pitaj` tool schema fields for budgets unless explicitly approved during implementation.

## Acceptance criteria

- Existing explicit `pitaj`, `model: "auto"`, `/pitaj <alias> ...`, `/pitaj snapshot ...`, and `/pitaj config ...` behavior remains backward compatible.
- Result presentation is deterministic and tested as a pure formatting boundary where practical.
- The answer remains prominent and first, with metadata and sidecar boundary notice rendered compactly after the answer.
- Auto-routed calls show resolved route/model information and routing reason when available.
- Snapshot calls show included/omitted context categories and bounded-context status without claiming full-branch review.
- Manual/no-context calls are distinguishable when metadata is available.
- The result block states the sidecar has no tools/file access when context/snapshot wording could be ambiguous.
- Usage tracking is in-memory and session-scoped by default.
- Budget thresholds are advisory warnings, not blockers.
- `/pitaj usage` displays current-session consult counts grouped by meaningful route/context categories.
- Any reset behavior is explicit and documented.
- `npm test` passes from `/home/quzma/.pi/agent/extensions/pi-pitaj`.
- A fresh-context reviewer checks the concrete implementation before close-out.

## Phase-boundary health

M3 leaves the project in a coherent state even if later M4/M5 work never lands: `pitaj` remains a focused sidecar consult tool, but its outputs become self-explaining and self-regulating. Later advisor shortcuts and full-branch mode can reuse the presentation and usage foundations rather than inventing their own observability path.

## Constraints and invariants

- Sidecar models remain tool-less and must not claim they inspected files unless caller-provided or snapshot context contains the excerpt.
- Result metadata must not expose secrets, auth headers, or unbounded raw context.
- Observability should be concise enough for routine tool responses.
- Budget warnings should guide behavior, not shame or block the caller.
- Prefer pure helper functions for formatting, usage classification, and warning policy so tests do not need a live Pi UI.
- Preserve TypeScript style: no `any`, no non-null assertions, type-only imports where applicable.
- Preserve current test command unless a task explicitly changes it: `npm test`.
- Do not add full-branch/session scraping while implementing usage or result presentation.

**Implementation context:** `index.ts`, `helpers.ts`, existing result details types, README/help text, tests.

### M3-T2 — Define usage event model and in-session store

**Depends on:** M3-T1
**Blocks:** M3-T3, M3-T4, M3-T5, M3-T6
**Can run in parallel with:** none during implementation; this fixes the accounting boundary.

**Implementation context:** `helpers.ts` or a focused usage module, `index.ts`, tests.

**Acceptance:**

- Defines a small structured usage event for each consult, including timestamp, requested model, resolved model/alias, route type, mode, brevity, risk, context source, approximate context chars, max output chars, success/error, and truncation flags where available.
- Adds an in-memory session-scoped store owned by the extension runtime.
- Records successful consults and handled consult errors without leaking prompt/context content.
- Classifies consults into useful groups such as low-risk/GPT-style, high-risk/Opus-style, explicit model, snapshot, and manual context.
- Keeps the model independent enough to be unit tested without live provider calls.
- Does not persist usage history to disk in M3.

### M3-T3 — Define advisory budget and warning policy

**Depends on:** M3-T2
**Blocks:** M3-T4, M3-T5, M3-T6
**Can run in parallel with:** docs drafting from M3-T6 after M3-T2, but final wording waits for policy implementation.

**Implementation context:** usage helper/module, settings helpers only if thresholds become configurable, tests, README/help text.

**Acceptance:**

- Defines default advisory thresholds for current-session usage:
  - GPT-style/low-risk consults: warn after 3 in the current session/reset window.
  - Opus-style/high-risk consults: warn after 3 in the current session/reset window.
  - Snapshot consults: warn after >= 5 snapshot calls in the current session/reset window; the default is documented in tests and README and may be calibrated in a later milestone.
- Treats warnings as advisory only; no consult is blocked.
- Explains what the caller should do when warned, such as prefer local verification, a reviewer/subagent, or a manual reset if starting a new task.
- Avoids magical task detection. Uses session-scoped counters plus optional explicit reset semantics.
- Adds deterministic tests for under-threshold, at-threshold, and over-threshold cases.

### M3-T4 — Add `/pitaj usage` summary command

**Depends on:** M3-T3
**Blocks:** M3-T5, M3-T6
**Can run in parallel with:** none; command behavior depends on usage and warning policy.

**Implementation context:** `index.ts`, `helpers.ts` or usage module, tests, README/help text.

**Acceptance:**

- `classifySpecialCommand()` and command routing recognize `usage` without treating it as a normal consult question.
- `/pitaj usage` displays a compact current-session summary: total consults, route/model group counts, snapshot/manual/no-context counts, warnings reached, and current budget status.
- `/pitaj usage reset` is included in M3. The command resets all in-memory session counters and confirms with a brief message such as `pitaj usage counters reset`. The summary display notes that counters also reset when the Pi session ends.
- Existing `/pitaj help`, `/pitaj aliases`, `/pitaj models`, `/pitaj check`, `/pitaj snapshot`, `/pitaj config`, and normal consult behavior remain unchanged.
- Tests cover command classification and summary formatting.
